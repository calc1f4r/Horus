## Vulnerability Title

Unnecessary Truncation When Calculating Fees

### Overview

See description below for full details.

### Original Finding Content

## Description

As reward balances are updated, Lido mints additional token shares representing a portion of rewards taken as a fee. While the calculation (defined in `Lido.distributeRewards()` line [574]) correctly accounts for the devaluing effect of minting additional shares, it includes more truncating division than necessary. In most cases, this results in slightly less StETH being distributed to fee recipients than intended by the Lido DAO. In particular, line [256]

```solidity
uint256 feeInEther = _totalRewards.mul(_getFee()).div(10000);
```

includes a division, and this result is used in the following calculation to determine `sharesToMint`. This issue is not directly exploitable and probably not a large discrepancy, hence the low risk rating. However, it may be possible for the unexpected discrepancy to affect the validity of any economic incentive analysis.

## Detailed Analysis

Although it is obvious that a single truncating division generally produces a smaller result than exact division, the impact is less clear when part of a larger calculation. As such, we ask: “What is the effect of the truncated `feeInEther` on the value of `sharesToMint`? Is it larger or smaller?"

For this, we introduce the following notation:

- `p`: the result of `_getFee()`, the staking reward fee rate (basis points).
- `r`: `_totalRewards` the amount of new rewards generated by Lido staking (ETH).
- `f`: `feeInEther` the concrete staking reward fee (ETH).
- `t`: the result of `_getTotalPooledEther()` (ETH). This includes the most recently reported balance changes and `_totalRewards`.
- `s`: `prevTotalShares` the total number of shares prior to this minting.
- `x`: `sharesToMint` the amount of new shares to be minted to correspond to the desired fee.

When accounting for truncated division, we have:

```plaintext
fQ = rp / 10000 (exact division)
fZ = ⌊fQ⌋ = ⌊rp / 10000⌋ (integer division)
```

> **Note:** Division in Solidity is integer division, where the fractional part (remainder) of the result is discarded (or truncated).

The current implementation of the `sharesToMint` calculation is thus most correctly expressed as:

```plaintext
xZ = ⌊fZ * s / (t - fZ)⌋
```

which is not the same as

```plaintext
xQ = ⌊fQ * s / (t - fQ)⌋
```

While we cannot directly compute `xQ` in Solidity using `fQ`, it can be done by rearranging the calculation such that the extra division is not required, assuming no overflow:

```plaintext
xQ = ⌊(r * p / 10000) * s / (t - (r * p / 10000))⌋
   = ⌊(r * p * s) / (10000 * t - r * p)⌋
```

With regards to impact, we now ask what is the relation between `xZ` and `xQ`?

## Proof (xZ ≤ xQ):

(Note all quantities are positive integers, so multiplication maintains the direction of any inequalities.) Since:

```plaintext
fQ - 1 < fZ ≤ fQ,
```

then

```plaintext
fZ * s ≤ fQ * s (× s)
fZ * s * (t - fZ) ≤ fQ * s * (t - fZ) (× (t - fZ) given t - fZ ≥ 0)
```

and

```plaintext
t - fQ ≤ t - fZ
fZ * s * (t - fQ) ≤ fZ * s * (t - fZ). (× fZ * s)
```

Thus,

```plaintext
fZ * s * (t - fQ) ≤ fZ * s * (t - fZ) ≤ fQ * s * (t - fZ)
fZ * s * (t - fQ) ≤ fQ * s * (t - fZ).
```

Rearranging, we have:

```plaintext
fZ * s / (t - fZ) ≤ fQ * s / (t - fZ)
```

and therefore:

```plaintext
xZ ≤ xQ ■
```

The truncated division produces a smaller result, so fewer fees are distributed.

## Recommendations

Consider avoiding the `div(10000)` in `Lido.sol:576` until later. Instead, perform the following calculation, which is otherwise equivalent but contains only a single integer division:

```solidity
sharesToMint = feeBasis * totalRewards * prevTotalShares / (10000 * newTotalPooledEther) - (feeBasis * totalRewards);
```

Here, the variable naming is kept consistent with `Lido.distributeRewards()`, and `feeBasis` is the result of `_getFee()`.

## Resolution

This issue was tracked in Issue #241 and resolved in PR #236. See changes to `Lido.distributeRewards()`.
